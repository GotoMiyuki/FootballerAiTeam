一、实验题目
基于LangGraph的足球运动员职业生涯多智能体协作系统——集成ReAct推理、多层记忆机制与多工具协同


二、实验内容
1.算法原理

本系统是一个面向足球运动员职业生涯管理的多智能体协作系统，核心架构基于LangGraph状态图（StateGraph）构建，融合了ReAct推理循环、Mission驱动的任务规划、多层记忆机制以及多工具协同调用。

1.1 多智能体协作架构

系统包含7个专业Agent，各司其职：
- Manager（总经理）：负责Mission创建、意图理解、任务分发与执行方向守护
- Coach（技能教练）：基于四维属性（进攻/防守/身体/守门）分析短板，借助RAG知识库制定训练计划
- Nutrition（运动营养师）：计算BMI/BMR/TDEE，制定个性化营养方案
- Analyst（表现分析师）：跨维度交叉分析属性不平衡、ACWR负荷监测、伤病风险评估
- Career（职业经纪人）：职业路径规划（career_planning）与转会分析（transfer_analysis）
- Reviewer（信息审查员）：在最终报告生成前审查各领域输出质量、检测Agent间冲突
- Document（报告生成官）：按mode（综合报告/公关声明/商业评估/媒体应答）生成最终产出

1.2 ReAct推理循环

每个专业Agent均继承自BaseAgent，内置ReAct（Reasoning + Acting）推理循环。其核心流程为：
（1）Thought（思考）：LLM接收任务prompt和球员档案，分析当前需要什么信息
（2）Action（行动）：LLM自主决定调用哪个工具（RAG检索/联网搜索/计算器/数据库读写），生成function_call
（3）Observation（观察）：工具执行结果以ToolMessage形式返回，追加到对话上下文
（4）迭代：LLM基于观察结果继续思考，判断是否需要更多工具调用
（5）Finish：LLM输出最终JSON结果，循环结束（最大迭代次数5次，防止死循环）

关键代码位于agents/base.py的_run_react_loop()方法：
- 使用llm.bind_tools(self.tools)将工具绑定到LLM
- 循环调用LLM，检测response.tool_calls决定是否继续
- 工具调用日志记录到_tool_call_log，追踪完整推理链

1.3 Mission驱动的规划方法

不同于传统的静态任务分解，本系统采用Mission-centric（以任务使命为中心）的规划方式：
（1）Mission Creation：Manager分析用户意图，创建包含核心目标（primary_goal）、产出类型（output_type）、目标受众（audience）、领域贡献配置（domain_contributions）的Mission对象
（2）动态路由：graph.py中的路由函数（route_after_manager、route_after_sub_agent等）根据Mission中的domain_contributions决定哪些Agent需要执行，按需动态调度
（3）Intent Checkpoint：所有领域Agent执行完毕后，Manager作为Intent Holder校验各领域输出与Mission核心目标的相关性，生成synthesis_guide供Document使用
（4）Replan机制：Manager Assess节点评估各领域输出质量（缺失/内容过短/冲突），若发现问题且plan_version < 3，触发Replan重新激活相关Agent
（5）Human-in-the-loop：在Document生成最终报告前设置中断点（interrupt_before=["document"]），允许用户审核中间结果并选择继续/重新规划/退出

1.4 信息审查（Reflection）

Reviewer Agent实现了类似Reflection模式的自我审查机制：
（1）静态检查层（零LLM成本）：检查needed=True的Agent是否全部产出、产出内容是否过短（<100字符）
（2）LLM深度审查层：检测Agent之间的建议冲突（如Coach建议高强度训练 vs Analyst警告伤病风险）、评估产出与Mission目标的相关性
（3）审查结果路由：通过（passed=True）→ 进入Document生成报告；不通过 → 路由回Manager Assess触发Replan；plan_version > 3时强制放行

2.关键代码展示

2.1 ReAct推理循环核心代码（agents/base.py）

```python
def _run_react_loop(
    self,
    task_prompt: str,
    player_profile: Dict[str, Any] = None,
    max_iterations: int = MAX_REACT_ITERATIONS,
) -> tuple[str, List[Dict[str, Any]]]:
    """执行 ReAct 推理循环。
    Thought(LLM推理) → Action(LLM选择工具) → Observation(工具执行结果)
    → Thought → ... → Finish(LLM输出最终答案)
    """
    self._tool_call_log = []
    messages = [SystemMessage(content=self.system_prompt)]
    
    if player_profile:
        profile_summary = self._format_player_context(player_profile)
        if profile_summary:
            messages.append(HumanMessage(content=profile_summary))
    
    messages.append(HumanMessage(content=task_prompt))
    llm_with_tools = self.llm.bind_tools(self.tools) if self.tools else self.llm
    
    for iteration in range(max_iterations):
        response = llm_with_tools.invoke(messages)
        messages.append(response)
        
        if hasattr(response, "tool_calls") and response.tool_calls:
            for tool_call in response.tool_calls:
                tool_name = tool_call.get("name", "")
                tool_args = tool_call.get("args", {})
                tool_id = tool_call.get("id", "")
                result_str = self._execute_tool(tool_name, tool_args)
                self._tool_call_log.append({
                    "agent": self.name,
                    "tool": tool_name,
                    "args": tool_args,
                    "result_preview": result_str[:300],
                })
                messages.append(ToolMessage(
                    content=result_str,
                    tool_call_id=tool_id,
                ))
        else:
            # 无工具调用 → 推理完成，输出最终结果
            return response.content, self._tool_call_log
    
    return messages[-1].content, self._tool_call_log
```

2.2 短期记忆机制（agents/base.py）

```python
class BaseAgent(ABC):
    def __init__(self, llm, tools=None, memory_size=None):
        self.llm = llm
        self.tools = tools or []
        self.memory_size = memory_size or config.SHORT_MEMORY_SIZE  # 默认5
        self._short_memory: List[Dict[str, Any]] = []  # 滑动窗口短期记忆

    def add_to_memory(self, message: Dict[str, Any]) -> None:
        """向短期记忆中追加一条消息，超出大小时自动滑动窗口。"""
        self._short_memory.append(message)
        if len(self._short_memory) > self.memory_size:
            self._short_memory = self._short_memory[-self.memory_size:]

    def get_context(self, messages, current_task=None):
        """构建上下文窗口。"""
        if current_task:
            self._short_memory = [{"role": "user", "content": current_task}]
        else:
            self._short_memory = messages[-self.memory_size:] if messages else []
        return self._short_memory
```

2.4 LangGraph图构建与动态路由（graph.py核心片段）

```python
def build_graph(agent_nodes, **kwargs):
    workflow = StateGraph(AgentState)
    
    # 添加节点
    workflow.add_node("manager", agent_nodes["manager"])
    workflow.add_node("manager_assess", agent_nodes["manager_assess"])
    workflow.add_node("reviewer", agent_nodes["reviewer"])
    # ... 其他Agent节点由AGENT_REGISTRY驱动动态添加
    
    workflow.set_entry_point("manager")
    
    # 条件边：Manager → 动态路由到需要的领域Agent
    workflow.add_conditional_edges("manager", route_after_manager, route_map)
    # 每个子Agent执行后 → 路由到下一个或Manager Assess
    for node_name in get_sub_agent_node_names():
        if node_name != doc_node:
            workflow.add_conditional_edges(node_name, route_after_sub_agent, route_map)
    # Reviewer → Document(通过) 或 Manager Assess(Replan)
    workflow.add_conditional_edges("reviewer", route_after_reviewer, route_map)
    
    memory = MemorySaver()  # LangGraph内置检查点记忆
    return workflow.compile(checkpointer=memory, interrupt_before=interrupt_before)
```

3.创新点&优化

（1）Mission-centric规划替代传统Task-centric：以Mission对象作为全局"北极星"，所有Agent的产出方向和Document的最终合成均以Mission.primary_goal为最高准则，避免了多Agent协作中的目标漂移问题。

（2）代码层预处理 + LLM推理双层架构：每个Agent在执行ReAct循环前先进行代码层预处理（如Coach的短板识别与属性不平衡检测、Analyst的ACWR计算与伤病风险评估、Career的市场估值与边际价值计算），将确定性计算留给代码，将需要语义理解和创造性推理的部分交给LLM，兼顾了准确性和灵活性。

（3）三层Prompt架构：Layer 1 Domain Identity定义Agent职责边界（稳定，所有mode共享）、Layer 2 Mission Context运行时动态注入当前Mission上下文、Layer 3 Execution Guide按mode选择执行指引，实现了Prompt的模块化复用和动态组合。

（4）Human-in-the-loop + Replan 闭环：在Document生成最终报告前设置中断点，允许用户审核中间结果。Reviewer发现问题时自动触发Replan（最多3次），形成"执行→审查→修正→再执行"的质量闭环。

（5）Agent Registry驱动的可扩展架构：新增Agent只需在registry.py中添加一个条目，graph.py的路由逻辑自动适配，无需手动修改图结构。

2. 评测指标展示及分析

（1）Agent调度准确性：Manager根据用户意图正确识别需要执行的领域Agent数量及优先级。通过意图分类决策树和易混淆场景速查表（如"转会传闻回应"→Document vs "转会可行性分析"→Career），准确区分相似但职责不同的场景。

（2）工具调用效率：每个Agent的ReAct循环中LLM自主决定工具调用次数和顺序。实际运行中，大多数任务在2-3轮工具调用内完成（如Nutrition先调计算器再输出方案，Coach先RAG检索再Search补充），最大迭代上限5次防止无限循环。

（3）Replan机制效果：Manager Assess + Reviewer双层质量保障，在Agent产出缺失或质量不足时自动触发Replan（最多3次），避免低质量输出进入最终报告。

（4）上下文管理效率：BaseAgent的滑动窗口短期记忆（默认5条）和Document的消息修剪机制（trim_messages_for_next_round）控制上下文长度，防止多轮对话中的Token爆炸和上下文污染。

（5）响应时间分析：代码层预处理（属性分析、ACWR计算等）耗时<0.1s，RAG向量检索耗时约1-3s（取决于知识库大小），LLM推理耗时约3-10s（取决于工具调用轮数）。整体单次请求端到端耗时约5-20s。
